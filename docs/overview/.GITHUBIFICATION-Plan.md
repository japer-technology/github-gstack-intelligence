# Githubification Implementation Plan — gstack

> Companion to [.ANALYSIS-Githubification.md](.ANALYSIS-Githubification.md). This document is the actionable implementation plan.

---

## Decisions Locked In

| Decision | Answer | Rationale |
|---|---|---|
| **AI Runtime** | `pi-coding-agent` (`@mariozechner/pi-coding-agent` from `badlogic/pi-mono`) | Multi-provider, headless/RPC mode, proven by GMI on GitHub Actions, session management built-in |
| **Distribution** | Back to upstream `garrytan/gstack` as an **optional feature** | Entirely additive — dot-prefixed directory, zero modifications to existing files |
| **This Repo** | Dev/reference repo — not a runtime execution environment | Holds source-of-truth skills, lifecycle scripts, tests |
| **Runtime Repos** | Private, with security layers as required | Execution repos are separate, managed per-org |
| **Workflow Model** | **Single unified `gstack-agent.yml`** | Routing logic lives in code (`router.ts`), not in YAML. One workflow file per execution repo. |

## Architecture: Two-Repo Model

### This Repo (`githubification-gstack`) — Dev Repo

Contains everything needed to develop, test, and publish the Githubification layer. All new files live in `.gstack-actions/` (dot-prefixed, cleanly separated from upstream gstack code).

### Execution Repos (user's private repos) — Runtime

Receive a single workflow file (`gstack-agent.yml`) and an optional configuration file. The workflow installs gstack-actions as a dependency and runs the unified agent.

```
┌──────────────────────────────────────────────────┐
│  Dev Repo (githubification-gstack)               │
│                                                  │
│  .gstack-actions/                                │
│  ├── lifecycle/     ← orchestration code         │
│  ├── skills/        ← CI-adapted skill prompts   │
│  ├── package.json   ← pi-coding-agent dependency │
│  └── tests/         ← skill + integration tests  │
│                                                  │
│  (upstream gstack code — UNTOUCHED)              │
│  review/, qa/, cso/, ship/, browse/, ...         │
└──────────┬───────────────────────────────────────┘
           │ publishes as npm package or
           │ GitHub Action composite action
           ▼
┌──────────────────────────────────────────────────┐
│  Execution Repo (user's private repo)            │
│                                                  │
│  .github/workflows/gstack-agent.yml  ← ONE file │
│  .gstack-actions/config.json         ← optional  │
│  .gstack-actions/state/              ← git state │
│                                                  │
│  (user's actual code)                            │
└──────────────────────────────────────────────────┘
```

## Phase 1 — Unified Lifecycle Engine

**Goal:** Build the three lifecycle scripts that power ALL skills through a single entry point.

### `.gstack-actions/lifecycle/indicator.ts`

GMI pattern. When a GitHub event triggers the workflow:

1. Add 🚀 reaction to the issue/PR/comment to signal "agent is working"
2. On completion, add ✅ or ❌ reaction
3. Uses `GITHUB_TOKEN` + `gh api` — no additional credentials

~30 lines. Stateless, called at the start and end of every run.

### `.gstack-actions/lifecycle/router.ts`

The brain. Maps GitHub event payloads to skill names. This replaces the multi-workflow approach — the routing logic lives in code, not in YAML.

```
Input:  GitHub event JSON (from $GITHUB_EVENT_PATH)
Output: { skill: string, context: SkillContext }
```

**Routing rules:**

| Event | Condition | Skill(s) |
|---|---|---|
| `pull_request.opened` | Always | `review` |
| `pull_request.opened` | If security files changed | `review` + `cso` |
| `pull_request.opened` | If UI/CSS files changed | `review` + `design-review` |
| `pull_request.synchronize` | Always | `review` (re-review) |
| `issues.opened` | Body contains `/skill-name` | Parsed skill |
| `issues.opened` | Label `office-hours` | `office-hours` |
| `issues.opened` | Label `investigate` | `investigate` |
| `issues.opened` | Label `design-consultation` | `design-consultation` |
| `issue_comment.created` | Body contains `/skill-name` | Parsed skill |
| `issue_comment.created` | Existing session for issue | Continue conversation |
| `schedule` (weekly cron) | — | `retro` |
| `schedule` (nightly cron) | — | `benchmark` |
| `release.published` | — | `document-release` |
| `deployment_status.success` | — | `canary` |
| `workflow_dispatch` | `skill` input parameter | Specified skill |

**Config override:** Router checks `.gstack-actions/config.json` in the execution repo:

```json
{
  "skills": {
    "review": { "enabled": true, "model": "claude-sonnet-4-20250514", "on_pr": true },
    "cso": { "enabled": true, "on_pr": "security-files-only" },
    "qa": { "enabled": true, "trigger": "issue-comment" },
    "retro": { "enabled": false },
    "benchmark": { "enabled": false }
  },
  "authorization": {
    "min_permission": "write"
  }
}
```

~100-150 lines. Pure logic, no side effects, easily testable.

### `.gstack-actions/lifecycle/agent.ts`

The executor. Takes the router's output and runs the skill using pi-coding-agent.

Responsibilities:

1. **Load the CI-adapted skill prompt** from `.gstack-actions/skills/{name}.md`
2. **Inject context** — PR number, diff, branch, repo name, issue body, prior conversation
3. **Invoke pi-coding-agent** in headless/RPC mode with the composed prompt
4. **Capture output** — the agent's response, any tool calls, any file modifications
5. **Post results** — format as Markdown, post as PR/issue comment via GitHub API
6. **Persist state** — commit session transcript to `.gstack-actions/state/` if multi-turn
7. **Handle errors** — post error summary, add ❌ reaction, exit non-zero

**Key integration with pi-coding-agent:**

- Use `@mariozechner/pi-agent-core` for programmatic invocation
- Configure tools: `read`, `write`, `edit`, `bash` (pi's built-in four)
- Add custom tools: `github_comment` (post to issue/PR), `screenshot` (Playwright)
- Session management: pi's tree-structured sessions map to issue-based conversations

~200-300 lines. The most complex piece, but well-scoped.

## Phase 2 — CI-Adapted Skill Prompts

**Goal:** Create CI-adapted versions of each skill, organized by priority tier.

### Adaptation Pattern

Each upstream `SKILL.md.tmpl` goes through this transformation:

| Original Element | CI Adaptation |
|---|---|
| `{{PREAMBLE}}` (local bash setup, session tracking, telemetry) | **Remove entirely.** CI context is injected by `agent.ts` |
| `{{BROWSE_SETUP}}` (connect to localhost daemon) | **Replace** with Playwright cold-start block |
| `{{BASE_BRANCH_DETECT}}` (detect main/master) | **Replace** with `$GITHUB_BASE_REF` from event payload |
| `AskUserQuestion` calls | **Replace** with "post comment and wait for reply" (multi-turn) or **remove** (single-shot) |
| Local file paths (`~/.gstack/`, `~/.gstack-dev/`) | **Replace** with `.gstack-actions/state/` (git-committed) |
| `$B` browse binary commands | **Replace** with Playwright API calls (see Phase 3) |
| Preamble tier metadata | **Retain** — used by router for priority/ordering |

### Tier 1 — Single-shot, event-driven (Phase 2a)

| Skill | Source | CI Complexity | Notes |
|---|---|---|---|
| `review` | review/SKILL.md.tmpl (~285 lines) | Low | Remove preamble, inject diff via `git diff $GITHUB_BASE_REF` |
| `cso` | cso/SKILL.md.tmpl (~622 lines) | Low | Remove preamble, pure code analysis, no browser needed |
| `retro` | retro/SKILL.md.tmpl | Low | Reads git log, generates summary, posts as new issue |
| `document-release` | document-release/SKILL.md.tmpl | Low | Reads release notes, updates docs, opens PR |
| `benchmark` | benchmark/SKILL.md.tmpl | Medium | Needs baseline comparison from committed state |

### Tier 2 — Needs browser or brief input (Phase 2b)

| Skill | CI Complexity | Notes |
|---|---|---|
| `qa` | High | Needs Playwright, screenshot capture, URL from issue body |
| `qa-only` | High | Same as qa but report-only |
| `design-review` | Medium | Needs screenshots of before/after UI changes |
| `canary` | Medium | Post-deploy monitoring, needs URL |

### Tier 3 — Multi-turn conversation (Phase 2c)

| Skill | CI Complexity | Notes |
|---|---|---|
| `office-hours` | Medium | Full conversation loop via issue comments |
| `plan-ceo-review` | Low-Medium | May ask 1-2 clarifying questions |
| `plan-eng-review` | Low-Medium | Architecture review, mostly single-shot |
| `investigate` | Medium | Root-cause debugging, may need multiple turns |
| `design-consultation` | High | Extended multi-turn design conversation |
| `autoplan` | Medium | Orchestrates CEO → design → eng sequentially |

### Not Adapted (remains local-only)

| Skill | Reason |
|---|---|
| `careful`, `freeze`, `guard`, `unfreeze` | Local safety — replaced by branch protection in CI |
| `browse` (standalone) | Interactive browser command — the *capability* is adapted, not the skill |
| `setup-browser-cookies` | Local cookie import — no CI equivalent |
| `setup-deploy` | One-time local config |
| `gstack-upgrade` | Local update — replaced by Dependabot/Renovate |
| `codex` | Multi-AI requires separate API key, high cost, low priority |
| `land-and-deploy` | Merge/deploy — partially covered by `ship` + `canary` |
| `ship` | Complex orchestration — defer to Phase 4 |
| `connect-chrome` | Local Chrome integration — no CI equivalent |

## Phase 3 — Browser Integration for CI

**Goal:** Replace gstack's persistent browse daemon with CI-compatible Playwright integration.

### Approach: Direct Playwright, Not Browse Binary

The browse binary (`browse/dist/browse`) is a compiled Bun binary optimized for interactive local use — persistent Chromium daemon, sub-second latency, cookie persistence across sessions. None of these properties matter on CI. Instead:

- **Playwright** is already a dependency of gstack (`playwright: ^1.58.2` in root `package.json`)
- Launch Chromium fresh per workflow run — cold start (~3-5s) is acceptable for CI
- Capture screenshots via Playwright API → upload to GitHub as issue/PR comment images
- No daemon, no port management, no token auth — just a library call

### Custom pi-coding-agent Tool: `browser`

Register a custom tool with pi-coding-agent that exposes browser capabilities:

```typescript
// Conceptual — tool registration for pi-coding-agent
{
  name: "browser",
  description: "Navigate URLs, take screenshots, interact with web pages",
  parameters: {
    action: "goto | click | screenshot | text | fill | ...",
    url: "string (for goto)",
    selector: "string (for click/fill)",
    value: "string (for fill)"
  },
  execute: async (params) => {
    // Uses shared Playwright browser instance
    // Returns text content, screenshot base64, or action result
  }
}
```

This gives the agent the same browser capabilities as the local `$B` commands but through pi-coding-agent's tool system.

### Screenshot Upload Pipeline

1. Playwright captures screenshot → saved to `/tmp/screenshots/`
2. Upload to GitHub via the API (`POST /repos/{owner}/{repo}/issues/{issue_number}/comments` with inline image)
3. Or use GitHub Actions artifact upload for large screenshot collections

## Phase 4 — The Single Unified Workflow

**Goal:** Create `gstack-agent.yml` — the ONE workflow file that execution repos need.

### Design

This is the only workflow file that matters for runtime execution repos. Everything else is in the code (router, agent, skills).

```yaml
name: gstack

on:
  pull_request:
    types: [opened, synchronize, reopened]
  issues:
    types: [opened, labeled]
  issue_comment:
    types: [created]
  release:
    types: [published]
  deployment_status:
  schedule:
    - cron: '0 6 * * 1'  # Weekly Monday 6 AM UTC (retro)
  workflow_dispatch:
    inputs:
      skill:
        description: 'Skill to run'
        required: true
        type: choice
        options:
          - review
          - cso
          - qa
          - retro
          - benchmark
          - office-hours
          - plan-ceo-review
          - plan-eng-review
          - investigate
          - design-consultation
          - document-release
          - autoplan
      target:
        description: 'Target (PR number, issue number, or URL)'
        required: false

permissions:
  contents: write
  pull-requests: write
  issues: write
  actions: read

jobs:
  agent:
    runs-on: ubuntu-latest
    concurrency:
      group: gstack-${{ github.event.pull_request.number || github.event.issue.number || github.run_id }}
      cancel-in-progress: true
    steps:
      - name: Indicate
        run: |
          # Add 🚀 reaction to signal "agent is working"
          # (indicator.ts logic — inlined or called via bun)

      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: oven-sh/setup-bun@v2

      - name: Install gstack-actions
        run: cd .gstack-actions && bun install --frozen-lockfile

      - name: Install Playwright browsers
        # Only for skills that need browser (router outputs needsBrowser flag)
        if: contains(fromJSON('["qa","qa-only","design-review","canary"]'), github.event.inputs.skill)
        run: bunx playwright install chromium

      - name: Authorize
        run: |
          PERM=$(gh api "repos/${{ github.repository }}/collaborators/${{ github.actor }}/permission" --jq '.permission')
          if [[ "$PERM" != "admin" && "$PERM" != "maintain" && "$PERM" != "write" ]]; then
            echo "::error::Unauthorized: ${{ github.actor }} has $PERM permission"
            exit 1
          fi
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Run agent
        run: bun .gstack-actions/lifecycle/agent.ts
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          GITHUB_EVENT_PATH: ${{ github.event_path }}
          GITHUB_EVENT_NAME: ${{ github.event_name }}

      - name: Commit state
        if: always()
        run: |
          cd .gstack-actions/state
          git add -A
          git diff --staged --quiet || {
            git commit -m "gstack: update state [skip ci]"
            git push
          }
```

### Why One Workflow Works

- **The router is code, not YAML.** The `router.ts` reads `$GITHUB_EVENT_NAME` and the event payload, determines which skill(s) to run, and passes them to `agent.ts`. Adding a new skill is a code change, not a workflow change.
- **Concurrency** is handled per-PR/issue with cancel-in-progress.
- **Conditional browser install** only happens for skills that need it (router outputs a `needsBrowser` flag).
- **Authorization** is a single step — same for all skills.

## Phase 5 — State Management & Multi-Turn Conversations

**Goal:** Implement git-as-memory for conversation persistence and result history.

### State Directory Structure (in execution repos)

```
.gstack-actions/state/
├── issues/
│   ├── 42.json                      # Issue #42 → session metadata
│   └── 57.json                      # Issue #57 → session metadata
├── sessions/
│   ├── 42-1711600000.jsonl          # Conversation transcript for issue #42
│   └── 57-1711600100.jsonl          # Conversation transcript for issue #57
├── results/
│   ├── review-pr-15-1711600200.md   # Review result for PR #15
│   └── cso-pr-15-1711600300.md      # CSO audit result for PR #15
└── benchmarks/
    └── baseline.json                # Performance benchmark baseline
```

### Multi-Turn Conversation Flow

For `/office-hours`, `/investigate`, and other conversational skills:

1. **Turn 1:** Issue opened → router detects skill → agent runs → posts response as comment → commits session to `state/sessions/{issue}-{timestamp}.jsonl`
2. **Turn 2:** User replies (issue comment) → workflow triggers → router loads existing session → agent continues conversation with full history → posts new comment → appends to session file
3. **Turn N:** Repeat until conversation reaches natural conclusion
4. **Closing:** When issue is closed, session is finalized (no more triggers)

### Session File Format (JSONL)

Each line is a JSON object:

```jsonl
{"role":"user","content":"I want to build a marketplace for AI skills","timestamp":"2026-03-28T22:00:00Z","source":"issue-body"}
{"role":"assistant","content":"Let me ask some clarifying questions...","timestamp":"2026-03-28T22:00:15Z","skill":"office-hours"}
{"role":"user","content":"We're targeting individual developers first","timestamp":"2026-03-28T23:10:00Z","source":"issue-comment-123456"}
```

## Phase 6 — Testing Strategy

**Goal:** Validate skills work correctly in CI without burning LLM credits on every test.

### Test Tiers

| Tier | What | Cost | Frequency |
|---|---|---|---|
| **1. Static validation** | Skill prompts parse correctly, router logic is correct, config schema validates | Free | Every commit |
| **2. Integration** | Lifecycle scripts run with mock GitHub event payloads, pi-coding-agent invokes correctly with mock LLM | Free | Every commit |
| **3. E2E smoke** | One skill (review) runs against a real PR with a real LLM | ~$0.15 | On PR, gated |
| **4. Full E2E** | All skills run against fixture repos | ~$2-4 | Weekly/manual |

### Test Files

## Phase 7 — Distribution as Optional Upstream Feature

**Goal:** Contribute back to `garrytan/gstack` as an opt-in feature.

### Contribution Strategy

Since `.gstack-actions/` is entirely self-contained (dot-prefixed, own `package.json`, no changes to existing code), the PR to upstream would be:

1. **Add `.gstack-actions/` directory** — all lifecycle, skills, and tests
2. **Add documentation** — `docs/githubification.md` explaining the feature
3. **Add to README.md** — one section: "Run gstack on GitHub Actions"
4. **No modifications** to any existing file (skills, browse, scripts, setup, package.json)

### What Upstream Gets

- Users who want local gstack: **nothing changes**
- Users who also want CI gstack: copy `gstack-agent.yml` to their repo's `.github/workflows/`, add `ANTHROPIC_API_KEY` secret, optionally add `.gstack-actions/config.json` for customization
- gstack maintainers: can evolve `.gstack-actions/skills/` alongside the original skills

### The `.gstack-actions/` Contract

The adapted skills in `.gstack-actions/skills/` reference but don't duplicate the original skills. They are transformations — the same quality standards, the same workflow structure, adapted for non-interactive CI execution. When upstream updates a skill, the CI-adapted version should be updated too (documented in CONTRIBUTING.md).

## Phase 8 — Security Layers (Execution Repo Concerns)

**Goal:** Document the security model for private execution repos.

Since execution repos are private with additional security layers, the plan accounts for:

| Layer | Implementation |
|---|---|
| **Authorization** | Workflow step checks collaborator permissions via `gh api` before any skill runs |
| **Secrets** | `ANTHROPIC_API_KEY` in GitHub Secrets — never in code, never in state |
| **State visibility** | `.gstack-actions/state/` is committed to git — in private repos this is fine; for public repos, sensitive conversations should use artifacts instead |
| **Concurrency** | `concurrency` group prevents parallel runs from stacking costs or race-conditioning state |
| **Cost control** | `config.json` allows disabling expensive skills, label-gating optional skills, model selection per skill |
| **Branch protection** | Replaces local `careful`/`freeze`/`guard` skills — GitHub-native branch protection rules and CODEOWNERS |
| **Token scope** | `GITHUB_TOKEN` uses minimal required permissions per the `permissions:` block |

## Key Architectural Decisions

1. **pi-coding-agent** as the AI runtime — headless/RPC mode, multi-provider, session management built-in
2. **Single unified workflow** (`gstack-agent.yml`) — routing is code, not YAML
3. **Playwright directly** (not browse binary) — cold-start acceptable for CI
4. **Git-as-memory** — state committed to `.gstack-actions/state/`
5. **Optional upstream feature** — entirely additive, no modifications to existing gstack code
6. **Dev repo vs execution repo** — this repo holds source of truth; execution repos get one workflow file + config
